标签化自动测试

  • 测试

支持功能

本工具主要实现了以下功能

  • 从DB中取得测试用例数据
  • 泛化执行JSF接口测试
  • 断言结果，使用鹏哥的统一断言类进行，主要包含CONTAIN,FULL,JSONPATH等断言类型
  • 展示测试报告
  • 支持多团队，可以满足不仅仅是B2B的需求，只需要在用例库中添加团队相关字段，很方便的就能扩展到其他团队
  • 支持接入Jenkins形成每日构建自动化测试

主要优点

框架和数据分离：

把测试用例直接写在Java工程中，这样做会带来一些问题：工程比较私人化，不便于团队其他同学熟悉；交接时需要交接一个一个的工程，不够方便；不够平台化，有种各自为战的感觉。后期肯定要使用Jenkins做持续集成的自动化测试，如果用例只放在JSON中，则每次对用例的小改动都需要进行一系列的Add、Commit、Push等操作，很显然不方便日常经常修改测试用例的情况
使用我们的框架就没有这样的烦恼了，由于框架和数据的分离，只需对数据进行修改即可，脚本每次会在数据库中读取最新的用例数据进行测试

支持标签化测试：

支持自定义标签去获得用例数据，比如可以定义一个测试计划只run某个接口服务的用例，或是可以run某个团队的所有用例，标签维度支持自己定义
使用标签化测试的好处有：

  • 灵活配置，支持标签化的测试，我们可以根据需求选择用例集中的一个标签，如图我们选择的标签为服务名，数据库中查出这28条用例，那么这个标签就会对这28条用例逐一进行测试

  • 同时，sql支持的灵活语法都可以运用在用例选择上，比如我们可以建一个测试计划只针对查询接口进行回归测试，所以我们可以使用sql如：
    SELECT * from case_info where method_name like ‘%query%’;
    进行查询，如下图，则只会查出方法名含有query的用例逐一进行测试

  • 支持报告展示：

引入reportNG，美化报告的展示，多标签时支持分组展示结果：

点击标题可以进入详情查看详细的接口调用信息，断言信息等

测试执行：

  • 可以右键直接执行想要执行的测试配置。
  • 可以使用maven命令执行测试：mvn clean test -U -Dxml=xmlFileName,通过参数来选择需要执行的xml文件。
  • 也可以使用Jenkins来实现定时构建测试。

待优化的点：

  • 自动化用例的前后置操作目前不支持，如增删改查数据库，发送MQ消息等
  • 测试报告目前只能手动查看，web页面展示和邮件发送目前还未支持
  • 用例数据仍是写死的内容，未来应该支持参数化的灵活配置，提高健壮性
  • 欢迎大家补充更多
