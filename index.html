jacoco on-the-fly环境搭建

安装软件

安装ant，安装jacoco：服务端 mkdir /apps，拷贝jacocoagent.jar到/apps。

修改tomcat配置

在服务端进入 vi /export/Domains/应用/server1/bin/start.sh
在末尾添加-javaagent:/apps/jacocoagent.jar=includes=*,classdumpdir=/export/lijuntao/invoiceSoa.ka.jd.com/classes,append=false,output=tcpserver,port=10002,address=192.168.195.149

参数解释

  • classdumpdir：存放jacoco自动取出的该应用的class文件
  • append：指定jacoco每次dump的时候是追加还是覆盖，由于我们需要统计的是每次自动化测试的独立的覆盖率，所以这里我们设为false
  • output:指定覆盖率统计结果输出方式，tcpserver就是按照配置的address和port输出到对应服务器
  • port：需要一个空闲端口号
  • address：IP地址，IP就是Tomcat服务器的机器的IP，至于是写服务器IP还是写127.0.0.1要看情况
    1. 如果是在Tomcat服务器上执行ant dump的话，就直接写address=127.0.0.1
    2. 如果执行ant dump不是在Tomcat服务器上执行的，就得写服务器IP（切记）

由于我们是在jenkins服务器上做dump，所以这里写服务器IP
（官方文档：http://www.eclemma.org/jacoco/trunk/doc/agent.html）

验证安装结果

重启tomcat，ps -ef|grep 10002查看jacoco配置是否加入成功，成功后我们则继续下一步

对比所需的class和java文件的处理

  • 利用jacoco自带的classdumpdir进行获取class
    如果用jacoco自带的classdumpdir参数来获取class的话，会获取到这个进程存在期间用到的所有class文件，包括引用的外部jar包里的。定义： If a directory is specified for this parameter the JaCoCo agent dumps all class files it processes to the given location。但是外部jar我们不需要去关心覆盖率，所以不建议使用这种方式

  • 把所需文件都拷贝出来放到一个独立的目录
    如果自己拷贝代码里的class文件，我发现有些问题， 拷贝结果比workspace里的少一些。原因是项目下有名称相同的class文件，cp时被覆盖掉了，所以cp的方法不准确，所以也不推荐这种方式

  • 配置sonar-project.properties
    可以直接将sonar.java.binaries配置为项目根目录workspace，sonar会自动去遍历其下的所有子模块中的calss文件，推荐直接使用这种方式，简单明了

  • 添加jacoco相关文件到项目
    在22上项目中,比如：/export/job/jenkins_cdldj/jobs/man-crm.jd.com(yao-crm-web)/workspace
    添加文件jacoco.sh/biuld.xml/sonar-project.properties
    这几个文件需按照各个文件中的注释配置你的参数（不清楚的可以单独联系我）
    sonar配置文件官方文档：https://docs.sonarqube.org/display/SONAR/Analysis+Parameters

配置jenkins持续集成方案

  • pre-steps
    由于需统计的是自动化测试的覆盖率，所以在执行用例前先dump一次，可以reset tomcat的覆盖率数据
    此步可以在jenkins中配置pre steps来做
  • 执行自动化测试
    这一步如果之前做过自动化测试的项目，那就直接保持不变即可
  • post-steps
    再次执行ant dump，导出本次自动化测试的覆盖率exec文件
    此步可以在jenkins中配置post steps来做
    建议直接使用shell脚本调用jacoco.sh来执行ant，因为post-steps之后可能会包含一些其他操作，集成到jacoco.sh中方便今后维护

覆盖率统计

根据exec文件和项目的class、java文件进行对比，统计覆盖率

  • 使用jacoco的mvn插件
    使用shell脚本执行mvn sonar的命令得出覆盖率结果，mvn方式执行出的jenkinsjob的名称会是:
    /usr/local/maven3/bin/mvn -f $path/pom.xml -e -B org.sonarsource.scanner.maven:sonar-maven-plugin:3.0.2:sonar -Dsonar.jacoco.itReportPath=$path/jacoco.exec -Dsonar.jacoco.reportPath=$path/jacoco.exec -Dsonar.host.url=http://192.168.179.22:9000

  • 使用sonar-scanner
    直接使用jenkins的sonar插件进行统计，这个由于自动化测试没法调用接口，又不像mvn命令可以手动触发，所以还没调通
    需要在post steps中配置sonar scanner，post-steps选择Execute SonarQube Scanner即可，将sonar-project.properties中的内容修改好后添加到Analysis properties配置中，JDK选择1.8，因为sonar 相关的东西对JDK要求高

  • 手动导出报告
    直接手动ant report导出html的报告，这种方法需要手动下载下来查看覆盖率，不够自动化，不建议使用

  • 使用jacoco的jenkins插件
    利用jenkins的jacoco插件配置好参数，进行查看，由于团队统一由sonar来管理代码质量和覆盖率，所以也不建议使用

  • 使用jacoco的jenkins插件
    利用jenkins的jacoco插件配置好参数，进行查看，由于团队统一由sonar来管理代码质量和覆盖率，所以也不建议使用

推荐使用1或2的方式，后面几种方式想了解的也可以联系我

踩过的坑

  • 坑1：
    使用sonar scanner是否必须要求class和source目录保持和源代码一致？
    扫描的目录是否必须要求结构和包名能对上？
    不需要，sonar verision在2.1之前的才需要

  • 坑2：
    是否可以把需要统计的java和class文件拷贝出来进行扫描？
    不合适，因为项目中总会存在不同路径下的同名文件，这样拷贝后就会覆盖掉一部分重复文件

  • 坑3：
    on-the-fly模式的jacoco会一直记录各种请求数据，如何保证统计的是本次自动化测试的请求，而不包含其他请求？
    首先我们来看其他请求的定义，我认为包含两种
    一是纵向的，上一次统计之后收到的所有请求都被错误统计，这个的解决方法是在运行自动化测试之前执行一次dump，dump后jacoco会自动reset JVM的统计数据
    二是横向的，本次自动化测试期间收到的来自其他系统或是测试人员手工请求的数据，这种方法目前没有特别好的避免方法，经过调研，腾讯的做法是写了java代码通过反射去调用jacoco的reset方法来进行的数据清理，在每个用例前的setup中reset，在teardown的时候dump，一个用例一个exec文件，最后将所有exec文件merge形成最终的覆盖率统计文件。但我认为如果把每天跑自动化的时间设置在早上八点或是之前，就可以避免手工测试的影响， 这一步可以作为以后的优化方向，优先级较低

  • 坑4：
    sonar-project.properties里的projectname是否可以配置为中文？
    网上大部分资料都说不可以。 后来自己尝试了一下，答案是可以写成中文的，但是要换一种方式，比如你想把项目名称命名为“测试”， 那么在这里就要写成“\u6d4b\u8bd5”，那么下面这个参数就应该这样写“sonar.projectName= \u6d4b\u8bd5”，说白了就是将中文转成Unicode。但是不建议这么使用，进来都用项目英文名就比较好了

spring-task-4.0.xsd文件加载方式问题

关于这个错误的问题，今天坑了我一天，后面终于发现问题了，记录一下：
服务在测试环境部署好后一直报错，访问也报500internal error，但是开发同学使用公司的Jone平台可以正常部署，也不会报错

出现这种问题后，我首先想到肯定是机器的原因，然后查了下xsd的加载方式

xsd加载方式：
spring在加载xsd文件时总是先试图在本地查找xsd文件(spring的jar包中已经包含了所有版本的xsd文件)，如果没有找到，才会转向去URL指定的路径下载。这是非常合理的做法，并不像看上去的那样，每次都是从站点下载的。

然后排查了下测试机器的外网访问权限，可以ping通外网

然后查到了网上一位老哥说的检查方式：

1，查找是否有对应版本的xsd文件。spring 默认会从 本地jar中获取该文件，找不到，则联网下载。一般该文件都会在包里面
打开jar包，spring-beans-4.0.xsd 文件在org.springframework.beans.factory.xml 下可以找到【其他的自己找】。
注意坑爹的问题也出来了，命名我的jar包中有这个文件，但是每次都是从网络地址获取，网络不行经常会报错，找不到原因。。。

2，查看该jar包下 META-INF 下 spring.schemas 文件中是否对应声明了，检查这儿的东西是不是和你的xml文件中一致。
然后搜，看到帖子说spring加载xsd的机制，是根据在 spring.schemas中声明的，也没怎么注意这个问题，然后一直找其他方面的问题，
最后没办法了，打开spring.schemas看看，结果tmd 这里面真没有4.0的声明信息，本以为最不会出错的地方，坑了我

总结：对于spring的xsd文件，在配置处最好不要带版本号，这样就会默认去找jar包里有的xsd文件，而不是找特定版本的

sql三张表问题

SQL三张表问题

今天遇到一个sql问题，暴露了自己平时只会写简单sql的能力。。ORZ
现记录在此，今后还是应该加强sql能力

学生表S id（学号） name age
科目表C id（科目号） name
成绩表G cid sid score
要求查询：名字姓张开头的学生的数学成绩，按年龄降序排序

方法一：纯select(效率低)
select S.name,G.score from S,C,G
where S.id=G.sid and G.cid=C.id
and S.name like ‘张%’ and C.name=’数学’ order by G.score desc
会在SCG的列数总和中去执行查询，这种方法执行的是笛卡尔积，效率低

方法二：使用子查询
select a.name,a.score,C.name
from (select S.name,S.age,G.cid,G.score from S,G where G.sid=S.id)a,C
where a.cid=C.id and a.name like ‘张%’ and C.name=’数学’
order by a.age desc
子查询会先去创建一张表a，然后外层再从a和C表中交叉查询出结果

方法三：使用left join关联
select S.name,G.score
from S left join G on S.id=G.sid
left join C on C.id= G.cid
where S.name like ‘张%’ and C.name=’数学’
order by G.score desc
关联相对子查询应该是效率更好的，因为不存在建表的过程

方法四：复合查询
select S.name,A.score
from S,A
where S.name like ‘张%’ and S.id in
(select G.sid from
G left join C on G.cid=C.id
where C.name=’数学’ )A
order by A.score desc

这些都只是一些写法而已，但是具体的效率还是要视情况而定，依然有许多不清楚的地方，比如在什么场合应该使用什么查询方式的选择问题，还有索引对于效率的影响问题都不太清晰。今后还是应该多加强复杂sql的学习。
